# ADR 075: OSC 52 クリップボードサポート

## ステータス

採用

## コンテキスト

bunterm はブラウザ経由でリモートターミナルを操作するが、ターミナル内で動作するアプリケーション（vim、tmux 等）がシステムクリップボードにテキストをコピーする手段がなかった。

### OSC 52 とは

OSC 52 は端末エスケープシーケンスの一つで、ターミナルアプリケーションからホストのクリップボードへの読み書きを可能にする。

```
ESC ] 52 ; c ; <base64-encoded-text> ST
```

- `c` はクリップボード選択（clipboard）
- テキストは Base64 エンコードで送信
- 多くのモダンターミナル（iTerm2, WezTerm, kitty 等）が対応

### ユースケース

- vim/neovim でのヤンク → システムクリップボードへのコピー
- tmux のコピーモードからのクリップボード転送
- CLI ツール（`pbcopy` 相当）からのクリップボード書き込み

## 決定

### OSC 52 クリップボードコピーの実装

ターミナル出力に含まれる OSC 52 シーケンスを検出し、ブラウザ側で Clipboard API を使用してシステムクリップボードに書き込む。

#### 処理フロー

1. サーバー側: PTY 出力から OSC 52 シーケンスを検出・パース
2. WebSocket 経由でクリップボードデータをブラウザに送信
3. ブラウザ側: `navigator.clipboard.writeText()` でクリップボードに書き込み

#### セキュリティ考慮

OSC 52 はターミナルアプリケーションが暗黙的にクリップボードへ書き込むため、セキュリティ上のリスクがある。

- **リスク**: 悪意のあるコマンド出力がクリップボードを上書きする可能性
- **緩和策**: ユーザーが設定で OSC 52 サポートを有効・無効にできる
- **ブラウザ制約**: Clipboard API は Secure Context（HTTPS）でのみ動作し、フォーカスされたタブでのみ書き込み可能

### 書き込みのみ対応

OSC 52 はクリップボードの読み取りも仕様上サポートするが、セキュリティリスクが高いため書き込み（コピー）のみ対応する。読み取り（ペースト）は通常のキーボード操作（Ctrl+V）に委ねる。

## 代替案

### OSC 52 非対応（手動コピーのみ）

**採用しなかった理由**:
- vim/tmux ユーザーにとってクリップボード連携は必須機能
- 他の Web ターミナル（ttyd, Wetty）も OSC 52 をサポートしている

### 読み取り + 書き込み両対応

**採用しなかった理由**:
- クリップボードの読み取りはセキュリティリスクが高い（パスワードマネージャの内容等が漏洩する可能性）
- ブラウザの Clipboard API の読み取りはユーザー操作（クリックイベント等）が必要で、自動読み取りが困難

## 影響

### Positive

- **ネイティブ体験**: vim/tmux でのコピー操作がローカルターミナルと同等に動作
- **設定可能**: セキュリティを重視するユーザーは無効化可能

### Negative

- **暗黙的クリップボード操作**: ユーザーが意図しないクリップボード上書きが発生する可能性
- **HTTPS 必須**: Clipboard API の制約により、開発環境の HTTP では動作しない場合がある
